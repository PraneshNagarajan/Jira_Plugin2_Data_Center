1. Babel Introduction

What is Babel and why do we need it?

Real-world analogy

Think of Babel like a language translator between countries.

Everyday Analogy

You write a letter in modern English (ES6). Babel translates it to a simpler form that old readers (old browsers) can understand (ES5).

Tech Analogy

You write JavaScript using modern syntax like const, arrow functions, and classes. Babel converts them to plain old var and function() that every browser understands.

concept

What is Babel?

Babel is a JavaScript transpiler — it takes modern JavaScript (ES6/ES2015+) and converts it into older JavaScript (ES5) that older browsers can run. It has several dozen individual libraries, but we need just 3 core ones.

why

Why do we need it?

Not all browsers support modern JavaScript features. By running code through Babel, your app works everywhere — from the latest Chrome to older browsers.

  • Converts const / let → var
  • Converts arrow functions → regular functions
  • Converts ES6 classes → prototype-based code
  Install command
# Install all 3 Babel modules at once
npm install --save-dev babel-loader @babel/core @babel/preset-env

3. .babelrc Configuration File

Telling Babel what rules to follow

Real-world analogy

Think of .babelrc like an instruction manual left on your desk. When Babel loads up, it looks for this file and reads: "Ah, I'm supposed to use the preset-env ruleset today."

step 1

Create the file

Create a file in your project root called .babelrc — note: no file extension, just the dot and the name.

  .babelrc
{
  "presets": ["@babel/preset-env"]
}
✅
When Babel runs, it automatically finds .babelrc, sees it should use @babel/preset-env, loads that module, and applies all its ES6→ES5 transformation rules across all your JavaScript files.
verify

How to verify it worked

After running npm run build, open bundle.js and search for your function. If Babel worked correctly:

  • const and let will be replaced with var
  • Arrow functions () => {} will be replaced with function() {}
  • No ES6 class syntax visible
  Before → After transformation
// BEFORE (your code — ES6)
const add = (a, b) => a + b;

// AFTER (babel output — ES5)
var add = function(a, b) { return a + b; };

4. CSS Loader — Method 1 (Inject into <style>)

css-loader + style-loader: inject CSS into the DOM

Real-world analogy

Method 1 is like a chef who cooks the sauce directly into the dish. The CSS ends up mixed into your JavaScript bundle, and JavaScript injects it into the page at runtime.

step 1

Install the two loaders

terminal
npm install --save-dev style-loader css-loader
step 2

Add the CSS rule to webpack.config.js

webpack.config.js
module.exports = {
  module: {
    rules: [
      {
        test: /\.css$/,
        // ⚠️ Order matters! Right to left execution:
        // 1. css-loader runs first  (reads CSS file)
        // 2. style-loader runs second (injects into DOM)
        use: ['style-loader', 'css-loader']
      }
    ]
  }
};
step 3

Import CSS in your JavaScript file

image_viewer.js
// Import CSS into your JS file (non-JS imports need file extension!)
import '../styles/image_viewer.css';
// Note: no variable assignment needed — just importing is enough
🔍
What happens under the hood:
1. css-loader reads the CSS file and converts its contents into a JavaScript string inside bundle.js
2. style-loader adds code to bundle.js that, when executed in the browser, creates a <style> tag and injects the CSS string into the <head>
📁

5. CSS Loader — Method 2 (Separate CSS File)

mini-css-extract-plugin: output a standalone style.css (Webpack 5 modern approach)

Real-world analogy

Method 2 is like a chef who plates the sauce separately. Your CSS ends up in its own style.css file, separate from the JavaScript bundle.

🚨
extract-text-webpack-plugin is dead for Webpack 5! It only supports Webpack 1–3. Installing it on Webpack 5 throws an ERESOLVE peer dependency error. Use mini-css-extract-plugin instead — the official modern replacement.
PluginWebpack SupportStatus
extract-text-webpack-pluginWebpack 1–3 only❌ Abandoned
mini-css-extract-pluginWebpack 4 & 5✅ Use this
css-minimizer-webpack-pluginWebpack 4 & 5➕ Optional: minifies CSS
step 1

Install mini-css-extract-plugin

terminal
# Core plugin — extracts CSS to a separate file
npm install --save-dev mini-css-extract-plugin

# Optional — also minifies the extracted CSS in production
npm install --save-dev css-minimizer-webpack-plugin
step 2

Update webpack.config.js

webpack.config.js
const MiniCssExtractPlugin = require('mini-css-extract-plugin');
const CssMinimizerPlugin = require('css-minimizer-webpack-plugin'); // optional

module.exports = {
  module: {
    rules: [
      {
        test: /\.css$/,
        use: [
          // MiniCssExtractPlugin.loader replaces 'style-loader'
          MiniCssExtractPlugin.loader,
          'css-loader'
        ]
      }
    ]
  },
  plugins: [
    new MiniCssExtractPlugin({
      filename: 'style.css'  // output filename
    })
  ],
  // Optional: minify the CSS output in production
  optimization: {
    minimizer: [
      '...',  // keeps Webpack's default JS minifier
      new CssMinimizerPlugin()
    ]
  }
};
step 3

Link the CSS file in your HTML

index.html
<link rel="stylesheet" href="build/style.css">
🔍
Loaders vs Plugins in Webpack:
Loaders pre-process files before they enter the bundle.
Plugins work outside the pipeline and can prevent files from entering the bundle entirely — which is how MiniCssExtractPlugin saves CSS to a separate file.

css-minimizer-webpack-plugin is a separate concern — it doesn't extract CSS, it only compresses already-extracted CSS. Use it on top of MiniCssExtractPlugin for smaller production output.
⚖️

6. Method 1 vs Method 2

Injected CSS vs Separate CSS file — which to choose?

Method 1 — style-loader (Injected)

Simple setup — just 2 loaders

Works automatically, no HTML changes needed

Great for component-scoped styles (React)

CSS loaded via JavaScript — slower initial render

CSS lives inside bundle.js — harder to cache separately

Violates traditional CSS file expectations

Method 2 — MiniCssExtractPlugin (Separate file)

Browser downloads CSS and JS in parallel — faster

Separate file can be cached independently

Follows traditional web conventions

Requires extra plugin and HTML link tag

Slightly more complex configuration

All CSS merged into one file (no component isolation)

7. Image Loader

url-loader + image-webpack-loader: smart image handling

Real-world analogy

Think of it like shipping packages. Small packages (tiny images) get stuffed directly into the delivery truck (bundle.js). Large packages are sent separately in their own truck.

step 1

Install both loaders

terminal
npm install --save-dev image-webpack-loader url-loader
step 2

Add image rule to webpack.config.js

webpack.config.js
{
  // Match jpg, jpeg, png, gif, svg files
  test: /\.(jpe?g|png|gif|svg)$/,
  use: [
    {
      loader: 'url-loader',
      options: {
        // 40000 bytes = 40KB threshold
        // < 40KB  → embed as Base64 in bundle.js
        // > 40KB  → save as separate file in build/
        limit: 40000
      }
    },
    'image-webpack-loader'  // runs FIRST (compresses image)
  ]
}
step 3

Import images in JavaScript

image_viewer.js
import big from '../assets/big.jpg';    // >40KB → separate file path
import small from '../assets/small.jpg';  // <40KB → Base64 data string

const img = new Image();
img.src = small;  // works as data URL
img.src = big;    // works as file path (needs publicPath fix!)
step 4

Fix large image paths with publicPath

Large images saved to the build folder need a publicPath prefix so the browser knows where to find them.

webpack.config.js — output section
output: {
  path: path.join(__dirname, 'build'),
  filename: 'bundle.js',
  // Prepend 'build/' to all file paths generated by loaders
  publicPath: 'build/'
}
Image sizeWhat url-loader doesResult in code
Small (< 40KB) Converts to Base64 string, embeds in bundle.js data:image/jpeg;base64,/9j/...
Large (> 40KB) Compresses and copies to build/ folder build/faa6a8f...jpg
⚠️
Loader order matters! Loaders in the use array run right to left. So image-webpack-loader (listed second) runs first to compress the image, then url-loader (listed first) runs second to decide where to put it.

8. Code Splitting

Load JavaScript only when it's actually needed

Real-world analogy

Imagine a restaurant menu. Instead of preparing every dish when the restaurant opens, the kitchen only starts cooking when a customer actually orders. Code splitting works the same way — only load the code users actually need, when they need it.

concept

How it works

Use System.import() (ES2015 dynamic import) to tell Webpack to split that module into its own separate bundle. Webpack automatically detects these calls during build and creates separate chunk files.

step 1

Basic code splitting — load on button click

index.js
const button = document.createElement('button');
button.innerText = 'Click me';

// onclick (lowercase!) - this is vanilla HTML, not React
button.onclick = () => {
  // System.import() returns a Promise
  // Webpack sees this and creates a separate chunk file
  System.import('./image_viewer')
    .then(module => {
      // module.default = the exported function/component
      module.default();
    });
};

document.body.appendChild(button);
step 2

Export a function from the split module

image_viewer.js
import small from '../assets/small.jpg';

// Export a function instead of executing immediately
export default () => {
  const image = new Image();
  image.src = small;
  document.body.appendChild(image);
};
🔍
What Webpack does behind the scenes: When Webpack sees System.import(), it automatically creates a separate 0.bundle.js file for that module. It also injects JSONP code into the main bundle.js to handle fetching chunks from the server dynamically.

9. Webpack Dev Server

Auto-rebuild, hot reload, in-memory serving

Real-world analogy

Without webpack-dev-server, you're like a sculptor who has to rebuild the entire statue every time you change one inch. With dev-server, it's like a live preview — touch one part and only that part updates instantly.

step 1

Install webpack-dev-server

terminal
npm install --save-dev webpack-dev-server@2.2.0-rc.0
step 2

Add serve script to package.json

package.json
{
  "scripts": {
    "build": "webpack --mode production",
    "serve": "webpack-dev-server"
  }
}
step 3

Start the dev server

terminal
npm run serve

10. React Code Splitting with React Router

Load routes on demand using getComponent + System.import

Real-world analogy

Imagine a large book. Instead of printing every chapter when someone buys it, you print chapters on demand — only when the reader asks for them. Routes = chapters. Code splitting = print on demand.

ℹ️
Strategy: Keep commonly visited routes (Home, ArtistMain) in the main bundle. Lazily load less-common routes (ArtistCreate, ArtistDetail, ArtistEdit) only when the user navigates to them.
before

Old approach — all routes loaded upfront (JSX style)

router.js — before
// All components imported at the top = loaded immediately
import ArtistCreate from './components/artists/ArtistCreate';
import ArtistDetail from './components/artists/ArtistDetail';
import ArtistEdit   from './components/artists/ArtistEdit';

// JSX-based routes (all loaded at once)
<Route path="/" component={Home}>
  <IndexRoute component={ArtistMain} />
  <Route path="artists/new" component={ArtistCreate} />
</Route>
after

New approach — plain routes with getComponent

router.js — after (code splitting)
// Remove the import statements for lazily loaded components!
// Only keep Home and ArtistMain (loaded upfront)
import Home from './components/Home';
import ArtistMain from './components/artists/ArtistMain';

const componentRoutes = {
  component: Home,
  path: '/',
  indexRoute: { component: ArtistMain },
  childRoutes: [
    {
      path: 'artists/new',
      // getComponent = lazy loading hook
      // location: current URL, cb: callback to pass component
      getComponent(location, cb) {
        System.import('./components/artists/ArtistCreate')
          .then(module => cb(null, module.default));
        //                              ↑ null = no error
        //                                       ↑ .default = actual component
      }
    },
    {
      path: 'artists/:id',
      getComponent(location, cb) {
        System.import('./components/artists/ArtistDetail')
          .then(module => cb(null, module.default));
      }
    },
    {
      path: 'artists/:id/edit',
      getComponent(location, cb) {
        System.import('./components/artists/ArtistEdit')
          .then(module => cb(null, module.default));
      }
    }
  ]
};

// Pass routes object to Router instead of JSX children
<Router history={history} routes={componentRoutes} />
⚠️
Cannot dynamically build the path string! Webpack scans your code statically for System.import() calls. The path must be a literal string — you can't generate it dynamically with string concatenation or variables.

11. Static Deployment Prep

Production optimizations before deploying

Real-world analogy

Before publishing a book, you proofread, remove filler words, and print the final edition. Production build does the same — removes debug checks, minifies code, and ships the leanest possible version.

step 1

Add DefinePlugin — set NODE_ENV for production React

React checks for process.env.NODE_ENV === 'production'. When set, React skips extensive error checking, making it significantly faster.

webpack.config.js
const webpack = require('webpack');

plugins: [
  // Defines a global variable in the browser scope
  // React reads this to skip development checks in production
  new webpack.DefinePlugin({
    'process.env.NODE_ENV': JSON.stringify(process.env.NODE_ENV)
  })
]
step 2

Update build script in package.json

Add NODE_ENV=production and the -p flag. The -p flag tells Webpack to minify all output — removing spaces, renaming variables, stripping comments (20–30% smaller files).

package.json
{
  "scripts": {
    // NODE_ENV=production → sets env variable
    // -p → minify output (20-30% smaller!)
    "build": "NODE_ENV=production webpack -p",
    "serve": "webpack-dev-server"
  }
}
OptimizationHowBenefit
NODE_ENV = productionDefinePlugin + env varReact skips dev-only checks → faster
Minificationwebpack -p flag20–30% smaller bundle size
Variable renamingAuto with -pShorter variable names = less bytes
Comment strippingAuto with -pAll comments removed from output
🚀
Quickest deployment — Surge.sh: After running npm run build, install Surge and run it in your build directory. Takes about 3 minutes to get live on the internet.
terminal — deploy with surge
# Build for production first
npm run build

# Install surge globally
npm install -g surge

# Deploy your build folder
surge ./build

12. package.json — Custom Webpack for Jira Plugin

Ditch CRA, use custom webpack for predictable output filenames

Real-world analogy

CRA is like an all-inclusive resort — great for standalone apps but too opinionated for embedding inside Jira. Custom webpack is like renting your own apartment — more setup, but full control over what goes where.

🚨
3 problems with a typical CRA-based package.json for Jira:

1. Two build systems fighting — react-scripts uses its own internal webpack. Your custom webpack.config.js loaders (babel-loader, css-loader etc.) are completely ignored.

2. React 19 + react-scripts 5 incompatibility — react-scripts@5.0.1 was built for React 17/18. React 19 breaks it with peer dependency errors.

3. Hashed/chunked output — CRA generates main.f855e6bc.js, 845.ee2883ba.chunk.js etc. Jira's velocity templates need fixed, predictable filenames like main.js.
❌ Problematic (CRA-based)

react-scripts in dependencies

React 19 breaks react-scripts 5

No webpack / webpack-cli

No @babel/preset-react (JSX broken)

url-loader redundant in Webpack 5

style-loader wrong for Jira plugin

✅ Fixed (Custom Webpack)

react-scripts removed entirely

React 18 — stable, widely supported

webpack + webpack-cli + webpack-dev-server

@babel/preset-react added for JSX

asset/resource replaces url-loader

MiniCssExtractPlugin for separate CSS

  package.json — clean custom webpack setup
{
  "name": "frontend",
  "version": "0.1.0",
  "private": true,
  "dependencies": {
    // React 18 — stable, widely supported, works with all tooling
    "react": "^18.2.0",
    "react-dom": "^18.2.0"
  },
  "scripts": {
    // webpack replaces react-scripts — predictable output filenames
    "build": "webpack --mode production",
    "start": "webpack serve --mode development",
    "test": "jest"
  },
  "browserslist": {
    "production": [">0.2%", "not dead", "not op_mini all"],
    "development": ["last 1 chrome version", "last 1 firefox version"]
  },
  "devDependencies": {
    // Babel core + presets
    "@babel/core": "^7.29.0",
    "@babel/preset-env": "^7.29.0",
    "@babel/preset-react": "^7.18.6",  // ← REQUIRED for JSX
    "babel-loader": "^10.1.0",

    // CSS handling
    "css-loader": "^7.1.4",
    "mini-css-extract-plugin": "^2.9.0",  // ← extracts to main.css

    // Webpack 5 core
    "webpack": "^5.105.0",
    "webpack-cli": "^5.1.0",
    "webpack-dev-server": "^5.0.0"

    // REMOVED: react-scripts, url-loader, image-webpack-loader,
    // style-loader — not needed with custom webpack 5 setup
  }
}
